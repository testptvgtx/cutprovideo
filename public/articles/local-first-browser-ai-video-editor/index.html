Local-first Architecture

把 AI 视频编辑器装进浏览器：本地优先架构的六个关键决策

浏览器 AI 的难点从来不只是“把模型加载进来”。真正的挑战，是把推理、时间线、缓存、渲染与失败恢复组合成一个可信的创作工具。

Timeline Studio 工程手记2026-07-17约 12 分钟

Timeline Studio 是一个 React 19 + Vite 6 构建的本地优先视频编辑器。语音合成、自动字幕、主体识别、数字人与视频导出都尽可能在用户设备内完成。本文不讨论“浏览器能不能跑 AI”，而是讨论如何让它跑得像一个产品。

01. 本地优先不是离线口号，而是一组架构约束

把媒体留在浏览器里，首先解决的是隐私和上传等待：用户选择的原始视频不必先经过服务端。但代价也会立刻转移到客户端——数百 MB 的模型下载、不同设备的内存上限、执行后端差异，以及浏览器编码能力的不一致。

因此“本地优先”的准确含义不是所有资源都内嵌在安装包里，而是：媒体与核心计算默认留在本机，模型按需从固定版本地址获取，重复使用时命中浏览器缓存，任何云端依赖都必须显式可见。

关键判断：把服务器成本移走，并不会让成本消失。它只是变成了下载、存储、内存、GPU 兼容性和更复杂的进度反馈。

02. 不追求“一个模型解决一切”

不同任务对模型、量化方式和执行后端的要求不同。Timeline Studio 选择多条专用路径：中文与多语种语音走 Piper/VITS ONNX，英文语音走 Kokoro 82M q8，字幕使用 Whisper small q8，视觉理解使用 YOLOS tiny 与 MODNet，而数字人管线由 JoyVASA 和 LivePortrait 组合完成。

这里最重要的选择是延迟加载。用户打开编辑器时，不应该同时下载字幕、数字人、视觉分割和所有语音模型。功能面板被真正使用时才创建对应运行时；同一个会话中则复用热会话，避免重复初始化 ONNX 图。

async function getDetector() {
  detectorPromise ??= pipeline("object-detection",
    "Xenova/yolos-tiny", { device: "wasm", dtype: "q8" });
  return detectorPromise;
}

03. Worker 不只是性能优化，而是产品边界

模型推理、音频解码、逐帧分析和编码任何一项都足以阻塞主线程。如果点击“生成字幕”后整个界面失去响应，那么即使模型最终给出正确结果，产品体验仍然是失败的。

项目把 ASR、视觉分析、数字人、画质增强和人声分离分散到独立 Worker。主线程只负责发送任务、接收结构化进度，并把完成结果转换成时间线资产。对大型 WebGPU 会话还需要串行创建，因为执行提供程序可能拒绝并发初始化。

Worker 同样提供了清晰的取消与错误边界：取消完整视频增强时，原片段保持不变；某个数字人关键帧出现非有限值或时间异常时，系统只重试该帧，并在必要时沿用上一张有效帧，而不是输出损坏纹理。

04. 时间线必须是 AI 结果的唯一归宿

很多 AI Demo 的终点是一个下载按钮。编辑器不能这样工作：生成的语音要插入播放头位置，字幕需要与语音绑定，重叠片段要自动进入新轨道，移动和删除时还要保持关联。

Timeline Studio 把 AI 输出统一变成 asset + clip。界面、预览与导出都读取同一份时间模型。生成语音时不会替换上一个结果，而是在播放头位置创建新的 voice clip；对应字幕共享关联 ID，并根据解码后的真实音频时长对齐。

voiceClip = {
  id, assetId, start, duration, volume,
  fadeIn, fadeOut, linkedCaptionId, captionSync: true
};

这个决定也解决了 WYSIWYG：字幕大小、智能裁切、贴纸、滤镜与转场不再有两份互相漂移的状态，而是在指定时间点解析同一个 clip 模型。

05. 视频 AI 的输出必须是时间数据

对视频第一帧做一次主体检测，然后把结果应用到全片，是最容易实现、也最容易出错的方案。人物一移动，智能裁切、字幕避让和背景移除都会立刻失真。

项目会以自适应间隔预分析视频，保存带时间戳的 YOLOS 几何与 MODNet 蒙版。预览和导出在当前源时间解析最近的分析记录。长视频扩大采样间隔，以可控的精度损失换取有上限的推理时间与内存。

temporalTrack = [
  { time: 0.0, box: A, matte: A1 },
  { time: 0.8, box: B, matte: B1 },
  { time: 1.6, box: C, matte: C1 }
];

06. 缓存要与版本绑定，导出要接受现实

服务工作线程对模型资源使用 cache-first，对应用壳使用 network-first。模型 URL 固定到不可变 revision，避免缓存中的图与运行时代码悄悄漂移。用户也可以主动清理模型缓存，因为“本地”不等于可以无限占用磁盘。

导出侧同样采用分层策略：浏览器原生支持目标格式时走快速路径；需要 MP4 转码时再加载 FFmpeg WASM；如果转码失败，保留已成功渲染的 WebM，不让用户因最后一步兼容性问题失去全部工作。

结论：浏览器 AI 是系统工程

模型准确率只是起点。一个可信的浏览器创作工具还需要明确的计算边界、可取消的 Worker、按需模型生命周期、时间化的视觉分析、统一的时间线语义、版本化缓存与保底导出。

这套架构不会让浏览器变成桌面工作站，但它证明了另一件更有价值的事：只要诚实地处理限制，复杂的 AI 媒体流程已经可以在一个链接里完成，并且不必默认交出用户的原始素材。
